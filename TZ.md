# Технічне завдання: AI Wallet

**Версія:** 1.0 · **Дата:** 2026-07-06 · **Статус:** чернетка для узгодження

---

## 1. Загальний опис

**AI Wallet** — non-custodial криптогаманець у форматі браузерного розширення з вбудованим AI-помічником. Ключова відмінність від MetaMask/Phantom: гаманець не просто показує сирі дані транзакції, а **пояснює простою мовою, що станеться**, попереджає про ризики перед підписанням і відповідає на питання про фінансову активність користувача в чаті.

### 1.1 Цільова аудиторія
- Початківці в криптовалютах, яких лякають hex-дані та `approve unlimited`.
- Досвідчені користувачі, яким потрібна аналітика витрат по всіх мережах в одному місці.

### 1.2 Ключові принципи
1. **Non-custodial**: приватні ключі ніколи не залишають пристрій користувача. Бекенд не бачить ні ключі, ні seed-фразу — тільки публічні адреси.
2. **AI пояснює, людина вирішує**: AI ніколи не підписує і не ініціює транзакції самостійно.
3. **Fail-safe**: якщо AI/бекенд недоступний, гаманець продовжує працювати як звичайний гаманець (з попередженням, що аналіз ризиків недоступний).

## 3. Архітектура

```
┌─────────────────────── Браузерне розширення (Manifest V3) ───────────────────────┐
│                                                                                   │
│  UI (React + TS)          Service Worker (TS)         Rust → WASM (crypto core)   │
│  • попапи, дашборд        • роутинг запитів dApp      • BIP-39/32/44 деривація    │
│  • екран підпису          • сесії, автолок            • підпис: secp256k1,        │
│  • AI-чат                 • кеш станів                  ed25519, ECDSA/BTC        │
│                                                       • AES-GCM + Argon2id        │
│  Content script + injected provider (EIP-1193, Wallet Standard)                   │
└───────────────┬───────────────────────────────────────────────────────────────────┘
                │ HTTPS/WSS (тільки публічні дані: адреси, unsigned tx)
                ▼
┌─────────────────────────── Бекенд (Rust) ─────────────────────────────────────────┐
│  API Gateway (Axum)                                                               │
│   ├─ balance-service    — агрегація балансів/цін (кеш Redis)                      │
│   ├─ tx-service         — декодування, симуляція, трансляція транзакцій          │
│   ├─ risk-engine        — rule-based скоринг + скам-списки                        │
│   ├─ ai-service         — OpenAI API: пояснення, чат із function calling          │
│   └─ indexer            — історія транзакцій, категоризація, аналітика           │
│                                                                                   │
│  PostgreSQL (історія, кеш метаданих)   Redis (ціни, rate limit, сесії)            │
└───────────────┬───────────────────────────────────────────────────────────────────┘
                ▼
  RPC-провайдери (Alchemy/Ankr — EVM, Helius — Solana, Bitcoin Core/mempool.space),
  CoinGecko, 4byte/Sourcify, скам-фіди, OpenAI API
```

**Ключове розділення:** усе, що торкається приватних ключів, — тільки в Rust/WASM усередині розширення. Бекенд оперує виключно публічними даними.

## 4. Технологічний стек

### 4.1 Розширення (клієнт)
| Компонент | Технологія | Обґрунтування |
|-----------|-----------|---------------|
| Фреймворк розширення | **WXT** + Manifest V3 | Сучасний DX, HMR, крос-браузерність (Chrome/Firefox/Edge) |
| UI | **React 18 + TypeScript**, Tailwind CSS | Екосистема, швидкість розробки |
| Стан | Zustand + TanStack Query | Легкі, добре лягають на service worker |
| **Крипто-ядро** | **Rust → WASM** (`wasm-bindgen`, `wasm-pack`) | Rust на клієнті: одна перевірена кодова база для ключів/підписів |
| Крипто-крейти | `bip39`, `bip32`, `k256`, `ed25519-dalek`, `bitcoin`, `alloy-signer`, `solana-sdk` (частково), `aes-gcm`, `argon2` | Аудитовані реалізації |
| dApp-провайдер | EIP-1193 / EIP-6963, Wallet Standard (Solana) | Сумісність з екосистемою dApps |

### 4.2 Бекенд (Rust)
| Компонент | Технологія | Обґрунтування |
|-----------|-----------|---------------|
| Веб-фреймворк | **Axum** + Tokio | Стандарт де-факто для async Rust, tower-мідлвари |
| БД | **PostgreSQL** + `sqlx` (compile-time перевірка SQL) | Надійність, JSONB для сирих транзакцій |
| Кеш | **Redis** (`fred`) | Ціни, rate limiting, кеш RPC |
| EVM | **Alloy** | Наступник ethers-rs, декодування ABI, RPC |
| Solana | `solana-client`, `solana-transaction-status` + Helius API | RPC + збагачена історія |
| Bitcoin | `bitcoin`, `bdk_wallet` + mempool.space API | UTXO, комісії, трансляція |
| OpenAI | `async-openai` | Chat Completions + function calling, streaming |
| Симуляція tx | Alchemy `simulateAssetChanges` / Tenderly API; Solana `simulateTransaction` | Зміни балансів до підпису |
| Спостережуваність | `tracing` + OpenTelemetry, Prometheus/Grafana | |
| Деплой | Docker, docker-compose (MVP) → Kubernetes (масштабування) | |
| CI/CD | GitHub Actions: `cargo clippy`, `cargo test`, `cargo audit`, e2e Playwright | |

### 4.3 AI-шар (OpenAI)
- **Модель для пояснень транзакцій:** `gpt-4o-mini` — дешево, швидко, достатньо для шаблонних пояснень.
- **Модель для чату/аналітики:** `gpt-4o` з function calling.
- **Вхід для пояснення:** уже декодовані структуровані дані (метод, суми, адреси, результат симуляції, ризик-фактори) — не сирий hex. Це різко підвищує якість і знижує токени.
- **Кешування:** однакові типи транзакцій (наприклад, стандартний ERC-20 transfer) пояснюються шаблоном без виклику API; AI викликається для нетривіальних випадків.
- **Ліміти:** rate limit на користувача, бюджет токенів на день, таймаут 10 с із fallback на rule-based пояснення.

## 5. API бекенду (основні ендпоінти)

```
POST /v1/balances            { addresses: {evm[], solana[], bitcoin[], tron[]} } → портфель
POST /v1/history             { address, chain?, cursor?, limit? }       → історія + описи
POST /v1/tx/params           { chain, from, token? }                    → nonce, gas, комісії, chain_id
POST /v1/tx/decode           { chain, raw_tx | tx_request }             → структурований розбір
POST /v1/tx/simulate         { chain, tx_request, signer }              → зміни балансів
POST /v1/tx/risk             { chain, tx_request, dapp_origin }         → {level, reasons[]}
POST /v1/tx/explain          { decoded, simulation, risk, lang }        → людське пояснення
POST /v1/tx/broadcast        { chain, signed_tx }                       → tx_hash
POST /v1/chat                { messages[], addresses[] } (SSE stream)   → відповідь AI
POST /v1/analytics/fees      { address, period }                        → витрати на комісії
POST /v1/analytics/summary   { address, period }                        → дашборд
GET  /v1/fees?chain=                                                    → оцінка комісій мережі
GET  /v1/prices?ids=                                                    → ціни
GET  /v1/health                                                         → статус сервісу
```

**Дані користувача — тільки в тілі POST.** Політика Chrome Web Store (код відхилення **Purple Copper**) забороняє передавати дані користувача в query-параметрах і HTTP-заголовках навіть по HTTPS: вони осідають у логах серверів, зворотних проксі й CDN, тобто дані витікають за межі шифрованого каналу. Адреса гаманця — це дані користувача, тож усі ендпоінти, що її приймають (`/history`, `/analytics/*`, `/tx/params`, `/balances`), — виключно `POST` з тілом JSON.

`GET` лишається **тільки** там, де параметр не є даними користувача і є публічним довідником: `/prices?ids=` (ідентифікатори монет CoinGecko), `/fees?chain=` (назва мережі), `/health` (без параметрів). Ці запити нічого не розкривають про власника гаманця, а GET дає HTTP-кешування.

Жодних адрес чи іншого користувацького контексту в заголовках запитів — розширення шле лише `Content-Type` і `Accept`.

Аутентифікація MVP: анонімна сесія, прив'язана до підпису повідомлення адресою (Sign-In-with-Ethereum-подібно) — без email/паролів на бекенді.

## 6. Безпека (критичні вимоги)

1. Seed/приватні ключі: **тільки** в зашифрованому `chrome.storage.local`; розшифрування — в пам'яті WASM на час операції; zeroize після використання.
2. Пароль → Argon2id (memory-hard параметри), ніколи не передається нікуди.
3. Бекенд **не має** ендпоінтів, що приймають ключі чи seed. Code review-правило + тест.
4. Content script ізольований від сторінки; повідомлення тільки через типізований `postMessage`-протокол з перевіркою origin.
5. Захист від prompt injection: дані транзакцій/dApp подаються в модель як структуровані поля в user-повідомленні, ніколи як інструкції; вихід AI — тільки текст пояснення, без керування гаманцем.
6. Rate limiting + CORS allowlist (тільки extension origin).
7. Залежності: `cargo audit` / `npm audit` у CI; фіксовані версії.
8. Перед релізом — зовнішній аудит крипто-ядра (WASM) і провайдер-протоколу.
9. Телеметрія без адрес за замовчуванням (opt-in), логи без PII.
